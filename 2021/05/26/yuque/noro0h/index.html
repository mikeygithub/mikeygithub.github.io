项目背景

作为反应式编程方向的第一步，微软在.NET 生态系统中创建了反应式扩展（Rx）库。然后 RxJava 在 JVM 上实现了反应式编程。随着时间的推移，通过 Reactive Streams 的工作，Java 标准化应运而生，这个规范为 JVM 上的 Reactive 库定义了一组接口和交互规则。它的接口已经在 Flow 类下集成到 java9 中。反应式编程范式通常是作为观察者设计模式的扩展而用面向对象语言提出的。您还可以将主要的反应流模式与熟悉的迭代器设计模式进行比较，因为在所有这些库中，Iterable 迭代器对具有双重性。一个主要的区别是，迭代器是基于 pull 的，而反应流是基于 push 的。
使用迭代器是一种必需的编程模式，即使访问值的方法完全由 Iterable 负责。实际上，何时访问序列中的 next（）项由开发人员来选择。在反应流中，上述对的等价物是 Publisher-Subscriber。但是发布者会在新的可用值出现时通知订阅者，这一推送特性是被动响应的关键。此外，应用于推送值的操作是以声明方式而不是强制方式表示的：程序员表示计算的逻辑，而不是描述其确切的控制流。
除了推送值之外，错误处理和完成方面也以定义良好的方式进行了讨论。发布者可以将新值推送到订阅服务器（通过调用 onNext），但也可以发出错误信号（通过调用 onError）或完成信号（通过调用 onComplete）。错误和完成都会终止序列。这可以概括如下：

项目简介

Reactor 是 JVM 的完全无阻塞的反应式编程基础，具有高效的需求管理（以管理“背压”的形式）。它直接与 Java8 功能 API 集成，特别是 CompletableFuture、Stream 和 Duration。它提供了可组合的异步序列 api —  通量（对于[N]元素）和单（对于[0 | 1]元素） —  广泛实现了反应流规范。Reactor 还支持与 Reactor netty 项目的非阻塞进程间通信。Reactor Netty 适合于微服务体系结构，它为 HTTP（包括 Websockets）、TCP 和 UDP 提供了支持背压的网络引擎。完全支持反应式编码和解码。

Reactor（反应式）编程是一种异步编程范式，与数据流和变化的传播有关。这意味着可以通过所使用的编程语言轻松地表示静态（例如阵列）或动态（例如事件发射器）数据流。

​

阻塞可能是浪费现代应用程序可以覆盖大量的并发用户，尽管现代硬件的性能不断提高，但现代软件的性能仍然是一个关键问题。大体上，有两种方法可以提高程序的性能：

  • 并行化以使用更多的线程和硬件资源。
  • 寻求更有效地利用现有资源。

通常，Java 开发人员使用块代码编写程序。在出现性能瓶颈之前，这种做法很好。然后是时候引入额外的线程，运行类似的阻塞代码了。但是这种资源利用率的扩展会很快引入争用和并发问题。更糟糕的是，封锁浪费了资源。如果仔细观察，只要程序涉及一些延迟（特别是 I/O，如数据库请求或网络调用），就会浪费资源，因为线程（可能有许多线程）现在处于空闲状态，等待数据。因此，并行化方法不是万能的。充分利用硬件的能力是必要的，但这也很复杂，而且容易造成资源浪费。

Reactor 特性

Reactor 项目的主要工件是 Reactor core，它是一个 Reactor 库，关注于反应流规范并以 java8 为目标。Reactor 引入了可组合的反应式类型，这些类型实现了 Publisher，但也提供了丰富的操作符词汇表：Flux 和 Mono。Flux 对象表示 0..N 项的反应序列，而 Mono 对象表示单个值或空（0..1）结果。
​

这种区别在类型中携带了一些语义信息，表示异步处理的粗略基数。例如，一个 HTTP 请求只产生一个响应，因此执行计数操作没有多大意义。因此，将这种 HTTP 调用的结果表示为 Mono比将其表示为 Flux更有意义，因为它只提供与零项或一项上下文相关的运算符。更改处理的最大基数的运算符也会切换到相关类型。例如，count 操作符存在于 Flux 中，但它返回一个 Mono。
